- 2026-06-04 (M1) вЂ” Entity ids = display names on import (teacher id "Bindu", subject id "ELGA", class id "Class 7") вЂ” names are documented unique; keeps legacy export stable and round-trip exact without a separate id map.

- 2026-06-04 (M1) вЂ” H8 (daily max) only fires where a CurriculumRequirement scopes the class+subject. The `maxPerDay` default 2 is a property of a requirement that omits it, NOT a global cap. Without this gate, freshly imported real data (empty requirements) would false-flag any legitimate 3Г—/day subject вЂ” undercutting the editor's credibility in M2. Coherent with H7's deferral.
- 2026-06-04 (M1) вЂ” H10 (pinned immovable) is NOT implemented in `domain/validate.ts`: it is a solver-loop invariant (does the solver move/remove a pinned placement?) with no static representation вЂ” `validate(project, timetable)` has no before-state to diff. Enforced in `solver/` (M3). The ROADMAP "H1вЂ“H6, H8вЂ“H10" grouping is loose, not a claim H10 is statically checkable.
- 2026-06-04 (M1) вЂ” H3 (block atomicity) implemented as block-definition integrity (length в‰Ґ 1, в‰Ґ1 class, в‰Ґ1 teacher). - 2026-06-04 (M1) вЂ” The M1 round-trip test is an EXACT-string match (`toBe`) against a canonical-format fixture written to the DATA_MODEL.md spec, NOT against real legacy-viewer output (no snapshot available). - 2026-06-04 (M2) вЂ” Edit operations (`domain/edit.ts`) are pure functions returning NEW arrays; undo/redo (`editorStore`) stores immutable snapshots (references, since nothing is mutated in place) в†’ `undo` restores the exact prior `Project` (verified by `toEqual`/`structuredClone`). Placements are identified by ref (activityId, day, period) so block occurrences on different days are independently movable, and any block cell carries the START-period ref в†’ dragging one ELGA cell moves all 15 atomically (it is one placement).
- 2026-06-04 (M2) вЂ” Two-store split: `projectStore` owns the Project + debounced (400 ms) IndexedDB autosave; `editorStore` owns UI state + history and commits through `projectStore`. Cross-store calls use `getState()`.
- 2026-06-04 (M2) вЂ” Live validation runs synchronously via a `useMemo` keyed on Project identity (a new object per commit). At VPPS scale `validate()` is well under the 16 ms budget, so no debounce is needed for correctness; debounce can be added in M-perf if profiling shows need.
- 2026-06-04 (M2) вЂ” Dev dep `fake-indexeddb` (justification: the only practical way to unit-test the "refresh restores from IndexedDB" AC and `persistence/db.ts` in Node/jsdom; test-only, not bundled). Runtime dep `@dnd-kit/core` is pre-approved ("dnd-kit", AGENTS.md В§1) for grid drag-drop.

- 2026-06-04 (M3) вЂ” A `CurriculumRequirement` is modeled as ONE canonical Lesson activity placed `periodsPerWeek` times (placements may repeat an activityId, exactly like the ELGA block on two days) вЂ” no distinct activity per period. `domain/requirements.ts` derives requirements + canonical lessons from an existing timetable; `normalizeProject` rebuilds a requirement-driven project. `makeSampleProject` returns the normalized sample so the Quota panel and solver have data.
- 2026-06-04 (M3) вЂ” CLASH-DEDUP FIX (oracle correctness): `validate` H1/H2 previously deduped occupancies by `activityId`, which silently missed two placements of the SAME canonical lesson (or two overlapping blocks) in one slot вЂ” the common case under the requirement model. Fixed to count raw occupancies (`occ.length > 1`); `derive.expandPlacement` de-dupes ids within a single activity so each placement contributes at most one occupancy per (entity, slot). Regression test added.
- 2026-06-04 (M3) вЂ” Solver (`solver/engine.ts`): seeded mulberry32 PRNG; blocks/pins frozen (H10); only missing requirement-periods are variables; dynamic most-constrained-variable selection (tie в†’ lowest index) with seed-shuffled value order в†’ deterministic per seed *provided the search completes within budget* (the fixture finishes in ~15 ms, far under the 5 s AC). `canPlace` mirrors every hard rule `validate` checks (incl. H6); the engine test still asserts the real `validate()` returns 0 hard on the output (honest oracle, not the solver's own bookkeeping). Teacher occupancy is counted once per (teacher, period), decoupled from class count, so blocks don't over-count.
- 2026-06-04 (M3) вЂ” Complete mode keeps ALL current placements fixed and fills only the shortfall; safe because the 30%-clear REMOVES placements (can't create a violation), so a known-good completion always exists. Min-conflicts/repair is deferred to M4. Generate mode (base = pinned only) is implemented but exercised in M4.
- 2026-06-04 (M3) вЂ” Cancel: in-engine via `shouldCancel()` (tested in Node); in the UI via `worker.terminate()` (the solve loop is synchronous inside the worker, off the UI thread). Solver budget uses `Date.now()` вЂ” affects only WHEN it stops, never the result when it completes, so determinism holds.
- 2026-06-04 (M3) вЂ” Soft scoring (S1вЂ“S6, default weights) implemented in `solver/score.ts` now (pure, needed for the worker `done` score and M4 candidate compare); wiring soft "amber" badges into the editor + the weight editor UI are M4 deliverables.
- 2026-06-04 (M4) вЂ” Candidates are RE-SCORED on the main thread with the live `SoftWeights` (`scoreTimetable` is pure + fast), so changing weights re-ranks instantly without re-solving.
