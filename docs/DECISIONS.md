# Decisions Log (append-only)

Format: `YYYY-MM-DD — decision — rationale`

- 2026-06-04 — New repo, separate from `timetable2025`; integration via legacy rawData export only — keeps the deployed viewer stable; the generator has different architectural needs.
- 2026-06-04 — Hand-rolled TS solver in a Web Worker, no CP libraries — problem size ≤ 504 cells; keeps bundle small, fully free, debuggable, deterministic.
- 2026-06-04 — ELGA modeled as an atomic `BlockActivity` (5 classes × 5 teachers × 3 consecutive periods) — matches reality: students regroup by English level across Classes 1–5; partial moves are meaningless.
- 2026-06-04 — Build order: editor with live validation first, solver second — schools trust a tool that catches mistakes before they trust black-box generation.
- 2026-06-04 — No backend/auth/sync in v1; IndexedDB + JSON file export — zero cost, offline-first, matches school's existing PWA philosophy.
- 2026-06-04 (M0) — Tailwind v3.4 with classic `tailwind.config.js` + `postcss.config.js` (not v4 CSS-first config) — v4's config model is churn we don't want in a deterministic build.
- 2026-06-04 (M0) — `build` script is `tsc --noEmit && vite build` (not bare `vite build`) — esbuild strips types without checking; this makes "strict, no any" actually enforced at build time. Paired with ESLint `@typescript-eslint/no-explicit-any` (strict catches implicit any; the rule catches explicit any).
- 2026-06-04 (M0) — Vitest default `environment: "node"`; UI tests opt into jsdom via `environmentMatchGlobs` (`src/ui/**`, `*.dom.test.tsx`) — keeps the domain/solver purity invariant (AGENTS.md §3) self-policing: impure DOM code can't silently pass in those layers.
- 2026-06-04 (M0) — Single `tsconfig.json` (dropped composite project references) — `tsc --noEmit` is incompatible with `composite`/`references` emit; one config is simpler and still typechecks vite.config.ts.
- 2026-06-04 (M0) — Dev deps beyond the AGENTS.md §1 pre-approved list, all standard for this stack: `vite`, `@vitejs/plugin-react`, `typescript`, `@types/*`, `@typescript-eslint/*`, `eslint` (+ react-hooks, react-refresh plugins), `jsdom`, `postcss`, `autoprefixer`, `@testing-library/jest-dom`. Each is required by the decided stack (ARCHITECTURE.md); no runtime/solver libraries added.
- 2026-06-04 (M1) — Entity ids = display names on import (teacher id "Bindu", subject id "ELGA", class id "Class 7") — names are documented unique; keeps legacy export stable and round-trip exact without a separate id map.
- 2026-06-04 (M1) — ELGA block detection keyed on the literal subject token "ELGA" (not a general multi-teacher-block heuristic) — it is the only block in VPPS today; a general heuristic risks false positives. One BlockActivity is created per distinct (length, class-set, teacher-order) signature, with one Placement per day-run, so multi-day ELGA reuses a single activity.
- 2026-06-04 (M1) — H8 (daily max) only fires where a CurriculumRequirement scopes the class+subject. The `maxPerDay` default 2 is a property of a requirement that omits it, NOT a global cap. Without this gate, freshly imported real data (empty requirements) would false-flag any legitimate 3×/day subject — undercutting the editor's credibility in M2. Coherent with H7's deferral.
- 2026-06-04 (M1) — H10 (pinned immovable) is NOT implemented in `domain/validate.ts`: it is a solver-loop invariant (does the solver move/remove a pinned placement?) with no static representation — `validate(project, timetable)` has no before-state to diff. Enforced in `solver/` (M3). The ROADMAP "H1–H6, H8–H10" grouping is loose, not a claim H10 is statically checkable.
- 2026-06-04 (M1) — H3 (block atomicity) implemented as block-definition integrity (length ≥ 1, ≥1 class, ≥1 teacher). True atomicity is structural in the data model — one Placement expands to all classes × periods × teachers via `derive.expandPlacement`, so partial placement is unrepresentable. The "can't move one class's ELGA" guarantee is editor behavior (M2), not a validate check.
- 2026-06-04 (M1) — The M1 round-trip test is an EXACT-string match (`toBe`) against a canonical-format fixture written to the DATA_MODEL.md spec, NOT against real legacy-viewer output (no snapshot available). A real-data round-trip will need a semantic compare (tolerant of blank lines / trailing spaces / ordering); the true end-to-end "paste into the legacy viewer" check is the M6 AC.
- 2026-06-04 (M2) — Edit operations (`domain/edit.ts`) are pure functions returning NEW arrays; undo/redo (`editorStore`) stores immutable snapshots (references, since nothing is mutated in place) → `undo` restores the exact prior `Project` (verified by `toEqual`/`structuredClone`). Placements are identified by ref (activityId, day, period) so block occurrences on different days are independently movable, and any block cell carries the START-period ref → dragging one ELGA cell moves all 15 atomically (it is one placement).
- 2026-06-04 (M2) — Two-store split: `projectStore` owns the Project + debounced (400 ms) IndexedDB autosave; `editorStore` owns UI state + history and commits through `projectStore`. Cross-store calls use `getState()`.
- 2026-06-04 (M2) — Live validation runs synchronously via a `useMemo` keyed on Project identity (a new object per commit). At VPPS scale `validate()` is well under the 16 ms budget, so no debounce is needed for correctness; debounce can be added in M-perf if profiling shows need.
- 2026-06-04 (M2) — Dev dep `fake-indexeddb` (justification: the only practical way to unit-test the "refresh restores from IndexedDB" AC and `persistence/db.ts` in Node/jsdom; test-only, not bundled). Runtime dep `@dnd-kit/core` is pre-approved ("dnd-kit", AGENTS.md §1) for grid drag-drop.
- 2026-06-04 (M2) — `gridModel.ts` (pure, no React) builds the grid view-model + per-cell conflict overlay, unit-tested in isolation; React components stay thin. Daily/weekly violations (H8/H9, period 0) are surfaced in the Conflicts panel, not as per-cell badges.
- 2026-06-04 (M3) — A `CurriculumRequirement` is modeled as ONE canonical Lesson activity placed `periodsPerWeek` times (placements may repeat an activityId, exactly like the ELGA block on two days) — no distinct activity per period. `domain/requirements.ts` derives requirements + canonical lessons from an existing timetable; `normalizeProject` rebuilds a requirement-driven project. `makeSampleProject` returns the normalized sample so the Quota panel and solver have data.
- 2026-06-04 (M3) — CLASH-DEDUP FIX (oracle correctness): `validate` H1/H2 previously deduped occupancies by `activityId`, which silently missed two placements of the SAME canonical lesson (or two overlapping blocks) in one slot — the common case under the requirement model. Fixed to count raw occupancies (`occ.length > 1`); `derive.expandPlacement` de-dupes ids within a single activity so each placement contributes at most one occupancy per (entity, slot). Regression test added.
- 2026-06-04 (M3) — Solver (`solver/engine.ts`): seeded mulberry32 PRNG; blocks/pins frozen (H10); only missing requirement-periods are variables; dynamic most-constrained-variable selection (tie → lowest index) with seed-shuffled value order → deterministic per seed *provided the search completes within budget* (the fixture finishes in ~15 ms, far under the 5 s AC). `canPlace` mirrors every hard rule `validate` checks (incl. H6); the engine test still asserts the real `validate()` returns 0 hard on the output (honest oracle, not the solver's own bookkeeping). Teacher occupancy is counted once per (teacher, period), decoupled from class count, so blocks don't over-count.
- 2026-06-04 (M3) — Complete mode keeps ALL current placements fixed and fills only the shortfall; safe because the 30%-clear REMOVES placements (can't create a violation), so a known-good completion always exists. Min-conflicts/repair is deferred to M4. Generate mode (base = pinned only) is implemented but exercised in M4.
- 2026-06-04 (M3) — Cancel: in-engine via `shouldCancel()` (tested in Node); in the UI via `worker.terminate()` (the solve loop is synchronous inside the worker, off the UI thread). Solver budget uses `Date.now()` — affects only WHEN it stops, never the result when it completes, so determinism holds.
- 2026-06-04 (M3) — Soft scoring (S1–S6, default weights) implemented in `solver/score.ts` now (pure, needed for the worker `done` score and M4 candidate compare); wiring soft "amber" badges into the editor + the weight editor UI are M4 deliverables.
